# Plan: Kubernetes-native vGPU Driver Image Builder (replace Terraform module)

> **Status**: v0.1.0 phases A–G complete. Live-cluster smoke surfaced 5 bugs
> (BUG-1 critical, BUG-2 high, BUG-3 med, BUG-4 low, BUG-5 info). See
> "Live-cluster smoke results" at the bottom of this document.

## Context

The original `vgpu-driver-builder` was a Terraform module (`main.tf`,
`variables.tf`, `outputs.tf` + `build/` assets) that, on `terraform apply`,
created a `gpu-driver-builder` namespace, S3 + registry secrets, ConfigMaps for
the Dockerfile and `nvidia-driver` script, and one or more one-off Kubernetes
`Job`s. The Jobs use `moby/buildkit:rootless` to build a Flatcar-compatible
NVIDIA vGPU driver image (using
`ghcr.io/andrewreid/flatcar-developer:<flatcar_version>-sources` as the base)
and push the result to a private registry. Two flavours exist: a
runtime-compile image (kernel modules built on each node by the DaemonSet) and
a precompiled image (modules baked in for a specific Flatcar release).

**Why change**:

- Terraform-driven flow forces a human to bump `flatcar_version` and re-`apply`
  whenever Flatcar publishes a release. Flatcar auto-updates nodes; the driver
  image lags behind.
- Terraform state + provider auth is an extra dependency for what should be an
  in-cluster operational concern.
- One-shot Job names embed input checksums, so they accumulate; there is no
  driver-image GC, no observability, no opinion about "which versions does the
  cluster currently need".
- NVIDIA GPU Operator auto-appends `-flatcar<VERSION_ID>` to `driver.version`,
  so the cluster *already declares* which (driver, Flatcar) image tags must
  exist. We can drive image builds from cluster state directly.

**Intended outcome**: a single Helm chart that installs a small Kubernetes
operator. The operator watches `Node` objects, learns which Flatcar releases
are in use, reads desired NVIDIA driver versions from a CRD, and ensures the
corresponding `<registry>/vgpu-driver:<driver>-flatcar<flatcar>` images exist
in the configured registry by spawning BuildKit `Job`s. No Terraform, no human
bump on Flatcar updates.

### Why an operator (vs CronJob, vs Job-per-version)

A reconciling controller is the right primitive because the desired state is
"for every (declared driver version) × (Flatcar version present on a Node), an
image exists in the registry". A CronJob can do this on a polling cadence, but
reacting to `Node` add/update events (when a node is upgraded by Flatcar's
auto-update) is the whole point of the redesign — a watch-based controller
does that natively. Operator pattern also gives us a CRD with `status` so
users can `kubectl get vgpudriverimages` and see what is built, building, or
failed.

### Why Python + kopf (vs Go controller-runtime)

The controller logic is orchestration-heavy, not performance-critical: list
Nodes, list registry tags, diff, create Jobs, update CRD status. `kopf`
provides watch-based handlers, finalizers, and CRD status patching with very
little code. A Go operator would be ~3–5x more code and require code-gen
tooling for marginal benefit here. Document the choice; Go is a valid
alternative if the team standardises on it later.

### Naming and tag scheme (locked by GPU Operator)

The GPU Operator appends `-flatcar<VERSION_ID>` to `driver.version` when it
pulls the driver image on a Flatcar node. The operator therefore **must**
publish images at:

```
<registry>/vgpu-driver:<driverVersion>-flatcar<flatcarVersion>
<registry>/vgpu-driver-precompiled:<driverVersion>-<kernelVersion>-flatcar<flatcarVersion>
```

Both modes ship in v0.1.0. Runtime-compile is the default; precompiled is
opt-in per-CRD via `spec.precompile: true`. Precompiled mode requires the
controller to also know each Node's `kernelVersion` (read from
`node.status.nodeInfo.kernelVersion` or NFD
`feature.node.kubernetes.io/kernel-version.full`), so the desired set becomes
`(driver, flatcar, kernel)` triples instead of `(driver, flatcar)` pairs.
Users wire GPU Operator with `driver.usePrecompiled=true` and the
`-precompiled` repository.

### Reconciliation logic (controller pseudocode)

```
on Node added/updated/deleted, on VGPUDriverImage changed, every 10 min,
  on Flatcar release-feed CronJob trigger:

    desired_node_set = { (flatcar_version, kernel_version)
                         for node in Nodes
                         if node matches spec.flatcar.nodeSelector }

    # Pre-emptive set from tracked Flatcar channels
    for ch in spec.flatcar.trackChannels:
        v = flatcar.latest_release(ch)
        k = flatcar.kernel_for_release(ch, v)
        desired_node_set.add((v, k))

    desired_drivers = spec.driverVersions

    if spec.precompile:
        desired = { (d, fc, k) for d in desired_drivers
                               for (fc, k) in desired_node_set }
        existing = registry.list_tags(spec.registry.repositoryPrecompiled)
        tag_fn   = lambda d, fc, k: f"{d}-{k}-flatcar{fc}"
    else:
        desired = { (d, fc) for d in desired_drivers
                            for (fc, _) in desired_node_set }
        existing = registry.list_tags(spec.registry.repository)
        tag_fn   = lambda d, fc: f"{d}-flatcar{fc}"

    missing = { t in desired if tag_fn(*t) not in existing
                              and not active_job_exists(t) }

    for t in missing:
        create_buildkit_job(*t, precompile=spec.precompile)
        record status.builds[t] = "Building"

    for job in jobs_owned_by(VGPUDriverImage):
        update status.builds[job.key] from job.status

    # --- Garbage collection (gated on spec.retention.enabled) ---
    if spec.retention.enabled:
        retained_flatcars = compute_retained_flatcar_set(
            current = {fc for (fc, _) in desired_node_set},
            tracked = {v.flatcarVersion for v in status.trackedChannelVersions},
            keep_previous = spec.retention.keepPreviousFlatcarVersions,
            all_known = sort_semver_desc(union of above + history))
        prunable = { tag for tag in existing
                     if extract_flatcar(tag) not in retained_flatcars
                     and tag_age(tag) > spec.retention.minAgeBeforeDelete }
        for tag in prunable:
            registry.delete_tag(tag)
            record status.pruned += {tag, reason, now}
            emit_event("Pruned", tag)
```

`compute_retained_flatcar_set` semantics: union of (a) every Flatcar version
present on a Node right now, (b) the latest version per tracked channel from
the poller, and (c) the next `keepPreviousFlatcarVersions` distinct versions
immediately below the highest current Node version, ordered by semver-aware
sort. Example: nodes on `4230.2.3`, tracked `stable=4230.2.4`, history
includes `4230.2.2`, `4152.2.0`, `4081.2.0`, with
`keepPreviousFlatcarVersions: 2` → retain `{4230.2.4, 4230.2.3, 4230.2.2,
4152.2.0}`; prune `4081.2.0`.

Tag age: read from registry manifest's `created` annotation if available;
fall back to `image.config.created` from the OCI manifest. Don't prune if age
is unknown.

`desired_node_set` derivation:

- Flatcar version: prefer NFD label
  `feature.node.kubernetes.io/system-os_release.VERSION_ID`; fallback parse of
  `node.status.nodeInfo.osImage`.
- Kernel version (precompiled only): prefer NFD label
  `feature.node.kubernetes.io/kernel-version.full`; fallback
  `node.status.nodeInfo.kernelVersion`.

Flatcar release-feed poller (in scope for v0.1.0):

- A `CronJob` runs the controller image with `--poll-flatcar` subcommand on a
  configurable schedule (default `@hourly`).
- The subcommand patches each `VGPUDriverImage`'s
  `status.trackedChannelVersions` with the latest version + kernel per
  channel listed in `spec.flatcar.trackChannels`.
- The main reconcile loop reads `status.trackedChannelVersions` (in addition
  to live Node data) when computing `desired_node_set`.
- Storing the poller's output in `status` (not `spec`) keeps the CRD spec
  immutable from the user's perspective and avoids spec/status reconcile
  loops.

## CRD spec

```yaml
apiVersion: vgpu.flatcar.io/v1alpha1
kind: VGPUDriverImage
metadata:
  name: default
spec:
  driverVersions:
    - "535.261.03"
    - "550.144.03"
  source:
    type: s3                          # or "http"
    uriTemplate: "s3://vgpu-drivers/NVIDIA-Linux-x86_64-${DRIVER_VERSION}-vgpu-kvm.run"
    credentialsSecretRef:
      name: s3-driver-storage-secret  # keys: S3_ENDPOINT_URL, AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY
  registry:
    repository: "harbor.example.com/gpu-drivers/vgpu-driver"
    repositoryPrecompiled: "harbor.example.com/gpu-drivers/vgpu-driver-precompiled"
    cacheRepository: "harbor.example.com/gpu-drivers/cache/vgpu-driver"
    authSecretRef:
      name: private-registry-secret    # dockerconfigjson
  flatcar:
    discoverFromNodes: true
    nodeSelector:
      nvidia.com/gpu.present: "true"
    trackChannels: ["stable"]
    arch: "amd64"
  precompile: false
  retention:
    keepPreviousFlatcarVersions: 2
    minAgeBeforeDelete: "168h"
    enabled: false
  build:
    buildkitImage: "moby/buildkit:rootless"
    resources:
      limits: { cpu: "4", memory: "8Gi" }
      requests: { cpu: "1", memory: "2Gi" }
    nodeSelector: {}
    tolerations: []
status:
  observedNodes:
    - { flatcarVersion: "4230.2.3", kernelVersion: "6.6.30-flatcar", nodeCount: 3 }
  trackedChannelVersions:
    - { channel: "stable", flatcarVersion: "4230.2.4", kernelVersion: "6.6.30-flatcar", observedAt: "..." }
  retainedFlatcarVersions: ["4230.2.3", "4230.2.2", "4152.2.0"]
  pruned:
    - { tag: "535.261.03-flatcar4081.2.0", reason: "outsideRetentionWindow", prunedAt: "..." }
  builds:
    - { driverVersion, flatcarVersion, kernelVersion, mode, tag, phase, jobName, lastTransitionTime, message }
  conditions:
    - { type: Reconciled, status: "True", reason: "AllImagesPresent" }
```

Conditions follow Kubernetes API conventions
(`type/status/reason/message/lastTransitionTime`).

## Out of scope for v0.1.0 (future work)

- Controller HA with leader election (single replica is fine for an
  orchestration controller; documented).
- Webhook for CRD validation beyond OpenAPI schema.
- ARM64 support (current Dockerfile is amd64-only).
- Multi-tenant isolation across `VGPUDriverImage` objects (v0.1.0 assumes one
  CRD per cluster; multiple are technically supported but share the operator
  namespace).
- Driver image signing (cosign) — straightforward follow-up once provenance
  labels land in B0.
- ECR/GAR token-exchange flows (users must pre-obtain a bearer token).
- True kopf integration tests (handlers tested at unit level only; e2e
  exercise deferred to live-cluster smoke).

  per-CRD via `spec.precompile`.
